# Research Knowledge Base Method

## Core Structure

The reusable pattern is not "collect more papers". It is "make the vault naturally support navigation, writing, intake, and cleanup."

The stable structure is:

- one canonical notes folder for mature notes
- one main index for storyline and navigation
- one page per research track
- one pending queue for intake
- one exclusion list for explicit out-of-scope material
- one audit page for quality control
- one browser/base view for filtering and regrouping
- shared attachment buckets for PDFs, figures, and web-harvest artifacts

## File Roles

### 1. Main Index

The main index does not store all details. It does four things:

- gives fast entry points
- gives task-based navigation
- gives a recommended reading order
- preserves a year-wise macro view of the field

### 2. Track Pages

Each track page should keep only:

- quick entry links
- recommended reading route
- core line
- bridge / strong-related papers
- backlog items

This is what makes the KB useful for related work writing and focused rereading.

### 3. Pending Queue

The pending queue must stay separate from the main index.

It distinguishes at least:

- local PDFs downloaded but not yet promoted
- online-confirmed papers not yet downloaded
- bridge papers that are useful but not in the main line
- note stubs that let you open the downloaded paper directly inside Obsidian instead of keeping the PDF orphaned

The pending page may also contain a managed auto-harvest block generated by `scripts/harvest_topic_papers.py`.

### 4. Audit Page

The audit page is not about academic ranking. It is for quality control.

Track:

- placeholder prose that was never customized
- notes still missing a core figure or results summary
- metadata anomalies
- notes that clearly need a rewrite

### 5. Browser/Base View

The browser is the search and regroup layer, not a replacement for the main index.

Use it to:

- filter by recent years
- regroup by task or method family
- locate a narrow slice quickly
- inspect recent updates

## Harvest-First Intake

When a new topic starts from the web, the first loop is:

1. Harvest candidate papers from multiple sources.
2. Merge duplicates and compare against the vault.
3. Review `core`, `bridge`, `low-confidence`, and `existing`.
4. Refine the field boundary.
5. Move only meaningful candidates into the pending queue.

This is better than waiting for a hand-collected PDF folder because it turns search into a repeatable artifact instead of a one-off memory exercise.

## Track Splitting Rules

Do not split by venue, year, or author cluster.

Split by how the user will actually use the literature:

- main method line
- adjacent task line
- data / generation / deployment branch
- application branch
- evaluation or robustness branch

Rules:

- start from questions the user will repeatedly ask
- each track must support a readable next-paper path
- a paper can serve multiple tracks, but one track should remain primary
- keep the core track count around `2-6`

## Canonical Note Rules

Do not use the canonical notes folder for shallow summaries.

The important distinction is:

- pending / triage material helps intake
- canonical notes replace a first re-read

Mature notes should keep a stable frontmatter and stable section order.

The frontmatter should keep at least:

- `title`
- `year`
- `venue`
- `tier`
- `subtype`
- `category`
- `official_url`
- `doi`
- `tags`
- `reading_status`
- `evidence_level`
- `one_sentence`

The body should keep a stable order so later scripts, browser views, and human review stay cheap.

A canonical note should answer, from the note alone:

- what problem the paper is solving
- what exact gap in prior work it targets
- what the method changes in operational terms
- what the training and inference flow look like
- why the method is supposed to work
- which evidence is strongest
- where the method is weak or still unclear

## Attachments and State

If the vault stores local attachments, use the same state buckets for PDFs and figures:

- `已入库`
- `待处理`
- `不相关`

For web discovery artifacts, prefer:

- `assets/paper_search/manifests`
- `assets/paper_search/reports`

The point is to make file state line up with note state.

## Maintenance Loop

Recommended long-term loop:

1. Run a web harvest for the topic or subtopic.
2. De-duplicate against the whole vault.
3. Put new candidates into the pending queue.
4. Download PDFs or keep official URLs.
5. Write mature notes only for papers that matter.
6. Update track pages and the main index.
7. Refresh the browser view and audit page.

## Quality Bar

Signals that a note is not mature yet:

- background and motivation are still generic filler
- the note never states in one sentence what the paper actually does
- there is no real method breakdown
- training and inference cannot be reconstructed from the note
- the note cannot explain why the claimed gain should happen
- there is no key figure or no core experimental takeaway
- the results section lists datasets but not the main comparison conclusion
- limitations and failure modes are missing
- the note does not explain how it relates to the current KB storyline
- the evidence boundary is not stated

A mature note should let the user decide within `30-60` seconds:

- whether the paper is worth deeper reading
- which track it belongs to
- what evidence is still missing

And within `2-3` minutes the user should be able to retell:

- the problem setting
- the core method path
- the main gain over the closest baseline
- the strongest supporting experiment
